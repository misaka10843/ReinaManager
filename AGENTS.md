# 交互与开发指南

> **权衡说明**：本指南偏向谨慎而非速度。简单任务自行判断。

## 0. 在开始之前先使用`$caveman`

## 1. 语言规范

- **交互语言**：所有思考、回答与交流必须使用**中文**。
- **代码注释**：代码中的注释必须使用**中文**编写。注释应侧重于解释复杂的业务逻辑或"为什么要这样做"，而非简单的代码翻译。

## 2. 思考先行

**不做假设。不隐藏困惑。暴露权衡。**

- 明确声明假设，不确定就问。
- 存在多种解读时全部列出，不默默选择。
- 有更简单方案就直言，该推回就推回。
- 不清楚就停下来，说清困惑点再提问。

## 3. 代码编写策略

### 3.1 文件管理（最小化新建）

- **优先修改现有文件**：在添加新功能时，首选在已有的相关文件中扩展代码。
- **慎重创建新文件/目录**：仅在现有文件过于庞大、职责不清，或新功能完全独立于现有模块时，才考虑新建文件或文件夹。

### 3.2 精准修改

**只动必须动的。只清理自己制造的混乱。**

- 不"改进"周边代码、注释或格式。
- 不重构没坏的东西。
- 匹配现有风格，即使你有不同偏好。
- 发现无关死代码：提一嘴，不删。
- 自己改动产生的孤儿（未使用的导入/变量/函数）：清理。
- 原有死代码：除非被要求，不删。
- **检验标准**：每行改动都应直接追溯到用户需求。

### 3.3 代码复用（DRY 原则）

- 编写新逻辑前，先检查项目中是否存在可用的函数、组件、类型定义或工具类。
- 严禁添加重复的辅助函数或类型定义，确保代码库精简与一致。

## 4. 代码质量与规范

### 4.1 简洁优先

**最小代码解决问题。不做投机性实现。**

- 不加未要求的功能。
- 不为单次使用写抽象。
- 不加未请求的灵活性/可配置性。
- 不为不可能场景写错误处理。
- 200 行能缩成 50 行就重写。
- 自问："高级工程师会觉得这过度设计吗？" 是就简化。
- 代码逻辑清晰、直观、优雅。
- 确保新代码融入现有架构风格。

### 4.2 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证目标：

- "加验证" → "写无效输入测试，使其通过"
- "修 bug" → "写复现测试，使其通过"
- "重构 X" → "确保重构前后测试通过"

多步任务列出简要计划：

```
1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
```

强成功标准可独立推进。弱标准（"先跑起来"）需反复确认。

### 4.3 工具链标准

- 本项目使用 **Biome** 进行代码格式化与 Lint 检查，使用 pnpm 包管理器。
- 本项目使用 **i18next-cli** 进行国际化字符串的检查和剔除无用键值对。
- 如果涉及国际化字符串的修改或添加，必须使用以下命令进行检查和剔除：
  ```bash
  pnpm i18n:status en-US --hide-translated # 还有 ja-JP 和 zh-TW，不需要 zh-CN
  pnpm i18n:sync # 剔除无用键值对
  ```
- 完成代码修改和国际化字符串检查后，按需验证代码规范和执行类型检查：
  ```bash
  pnpm check # 该指令已经包含了代码格式化、Lint 检查和类型检查
  ```
- 如果涉及 rust 后端的修改，按需使用以下命令进行格式化和检查：
  ```bash
  cd src-tauri
  cargo fmt
  cargo clippy
  ```

### 4.4 命令执行规范

- 执行 Shell 命令时，**尽量一条一条执行**。
- **慎重**使用 `&&`、`;` 或 `|` 将多条命令合并在同一个调用中。确保每步操作的输出都能被独立捕获和分析。

---

**指南生效标志**：diff 中不必要的变更更少，因过度设计导致的重写更少，澄清性问题在犯错前而非犯错后提出。
